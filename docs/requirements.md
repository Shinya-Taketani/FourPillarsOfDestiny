# requirements.md

## 1. 文書概要

本書は、KDE Plasma 向けに開発する阿部泰山系・泰山流四柱推命ソフトウェア **FourPillarsOfDestiny** の初版要件定義書である。

本プロジェクトは、単なる命式表示ツールではなく、以下を満たすことを目的とする。

- 生年月日・出生時刻・出生地等から命式を正確に算出できること
- 初心者でも読みやすい命式表示を提供できること
- 研究者・実務者向けに、力量関係・用神候補・行運評価などの詳細情報を段階的に表示できること
- 一般四柱推命の共通計算基盤と、泰山流固有の評価ルールを分離し、将来的なルール差し替えや拡張に耐えられること

本書では、**既知事項**、**仕様上の仮定**、**要確認事項**を分離して記述する。

---

## 2. 背景と前提

### 2.1 背景

本プロジェクトは Linux / KDE Plasma 環境に親和的なデスクトップアプリとして設計する。
UI は Qt6 + KDE Frameworks 6 + Kirigami を前提とし、軽量性、保守性、将来的な拡張性を重視する。

### 2.2 既知事項

現時点の調査結果から、阿部泰山系文献群として以下のような役割分担が示唆されている。

- 万年暦・命式算出
- 鑑定法
- 奥義的理論
- 調候用神
- 行運看法
- 実践鑑定
- 古典注解

また、緒方泰州系教材では、次のような論点が確認できている。

- 命式の出し方
- 通変星
- 十干・十二支の実践的力量関係
- 判断法
- 用神
- 実践鑑定論

このため、ソフトウェア要件としては、**算出機能**だけでなく、**評価機能**と**根拠表示機能**を持つ構成を前提とする。

### 2.3 仕様上の仮定

- 一般四柱推命として広く共通する部分は、共通計算基盤として先行実装できるものとする
- 泰山流らしさは、主に「力量評価」「調候用神」「行運評価」「実践判断」に現れるものとして設計する
- 流派固有ルールのうち原典確認が済んでいないものは、断定実装せず「要確認ルール」として別管理する

### 2.4 要確認事項

以下は初版時点では確定しない。

- 格局と調候の正式な優先順位
- 用神の正式決定手順
- 力量数値化の正式計算式
- 月律分野蔵干の厳密な適用方法
- 接木運の詳細処理
- 極意秘密皆伝に相当する特殊ルール
- 滴天髄和解由来の高度解釈ルール

## 3. ソフトウェアの目的

本ソフトの目的は、阿部泰山系・泰山流四柱推命をベースに、以下の三用途に対応する統合アプリケーションを提供することである。

1. **学習用途**  
   命式の基本構造、通変星、十二運、大運・流年を理解しやすく提示する。

2. **研究用途**  
   力量関係、通根、透干、格局候補、用神候補、行運との関係などを比較検討できる。

3. **鑑定実務用途**  
   保存済み人物、鑑定履歴、レポート出力、再計算、比較、印刷を扱える。

- 文献ベースで理論を比較したい利用者
- 一般四柱推命と泰山流の評価差分を見たい利用者

### 4.3 鑑定実務者

- 実務で命式を継続的に扱う利用者
- 人物情報の保存、履歴管理、レポート出力を必要とする利用者
- 命式結果だけでなく、判断根拠のトレースを必要とする利用者

## 6. コア機能要件

### 6.1 命式入力機能

#### 目的
出生情報を受け取り、命式算出の前提データを構築する。

#### 入力項目
- 生年月日
- 出生時刻
- 性別
- 出生地
- タイムゾーン
- 出生時刻不明フラグ
- 地方時補正の適用有無
- サマータイム考慮の適用有無
- 暦補正設定

#### 要件
- 日付入力は必須とする
- 時刻は任意入力とし、未入力時は時柱未確定として扱う
- 入力値は内部で正規化する
- 入力エラーは UI 上で明示する

#### バリデーション
- 存在しない日付は拒否する
- 時刻形式不正は拒否する
- 出生地が未設定でも算出可能とするが、地方時補正を無効化する

---

### 6.2 命式算出機能

#### 目的
入力情報から四柱推命の基本構造を算出する。

#### 算出対象
- 年柱
- 月柱
- 日柱
- 時柱
- 蔵干
- 通変星
- 十二運
- 五行分布
- 大運
- 流年

#### 要件
- 月柱算出は節入り基準で行う
- 時刻不明時は時柱算出を保留し、三柱ベース結果を返せること
- 算出ロジックは UI から独立した純粋関数中心で構成すること
- 命式算出結果は構造化されたドメインモデルとして返却すること

#### 出力要件
- 算出結果には使用した補正条件をメタデータとして含める
- 再現性確保のため、入力値とルールバージョンを保持できること

---

### 6.3 初心者向け結果表示機能

#### 目的
命式の全体像を、難解な専門用語に偏りすぎずに表示する。

#### 表示内容
- 四柱表
- 通変星
- 十二運
- 五行分布
- 大運表
- 流年表
- 簡易要約

#### 要件
- 表形式と説明文を併記すること
- 色だけに依存しない識別手段を持つこと
- 印刷しやすいレイアウトにできること
- 初心者向け表示では未確定ルールを断定表現しないこと

---

### 6.4 上級者向け詳細表示機能

#### 目的
泰山流研究・実務向けに、評価根拠と候補を詳細表示する。

#### 表示対象
- 力量関係
- 通根評価
- 透干評価
- 偏枯傾向
- 格局候補
- 用神候補
- 行運との相互作用
- 根拠ログ
- ルール信頼度

#### 要件
- 候補と確定を分けて表示すること
- 「確定 / 有力 / 仮説 / 要確認」のような信頼度表示を持つこと
- どのルールがどの判断に影響したか追跡できること
- 将来的なルール更新に対応できるよう、表示データはルールエンジン依存の DTO として受け取ること

---

### 6.5 保存・履歴管理機能

#### 目的
人物情報・鑑定結果・再利用情報をローカルに保存する。

#### 保存対象
- 人物プロフィール
- 入力原データ
- 算出結果キャッシュ
- 評価結果
- レポート出力履歴
- ユーザー設定

#### 要件
- ローカル DB は SQLite を使用する
- 個人情報はローカル保存を前提とする
- ルールセットやアプリバージョンを保存し、再現可能性を担保する
- キャッシュと元データは分離して保存する

---

### 6.6 レポート出力機能

#### 目的
鑑定結果を配布・印刷・保管しやすい形式に出力する。

#### 出力形式
- PDF
- テキスト
- JSON
- 画像（命式表）

#### 要件
- 初心者向けレイアウトと詳細レイアウトを切り替えられること
- 印刷プレビューに対応すること
- 将来テンプレート差し替えができる構成にすること
- 日本語組版を考慮したレイアウト余白を持つこと

## 8. 非機能要件

### 8.1 性能
- 単一命式の算出はローカルで即時応答を目指す
- 大運・流年を含む詳細表示でも UI スレッドを長時間ブロックしない
- 再計算は差分更新またはキャッシュ利用を考慮する

### 8.2 保守性
- UI と計算ロジックを分離する
- 一般四柱推命ロジックと泰山流ルールを分離する
- テーブル駆動または設定ファイル駆動を優先する
- ルール差し替え時に UI を大きく変更しなくて済む構造にする

### 8.3 可読性
- ドメイン用語に沿った型名・クラス名を用いる
- コメントは日本語で記述する
- ルールの出典・信頼度・適用条件をメタデータとして保持する

### 8.4 テスト容易性
- 計算ロジックは純粋関数中心とする
- 既知の命式サンプルとの照合ができること
- 境界時刻、節入り直前直後、時刻不明などのテストケースを持つこと
- 泰山流固有の未確定ルールは tentative 扱いでテスト分類すること

### 8.5 移植性
- 対象 OS は Linux を主とする
- KDE Plasma との親和性を優先する
- 依存は Qt6 / KDE Frameworks 6 / CMake を中核とする

### 8.6 セキュリティ・個人情報配慮
- 個人データはローカル保存を基本とする
- 将来的な暗号化拡張が可能な設計とする
- 不要な外部通信を行わない
- バックアップ・エクスポート時の扱いを明示する

## 9. 技術要件

### 9.1 推奨技術スタック
- 言語: C++20
- UI: Qt6 + QML + Kirigami
- KDE 連携: KDE Frameworks 6
- ビルド: CMake
- ローカル DB: SQLite
- テスト: Qt Test + Catch2 もしくは doctest

### 9.2 採用理由
- KDE 向けデスクトップアプリとして自然な構成である
- 型安全性と長期保守性を確保しやすい
- ルールエンジンと UI を分離しやすい
- Linux 上での配布・運用が容易である

- レポート出力ユースケース
- 設定更新ユースケース

#### Domain 層
- 入力モデル

- 通変星モデル
- 十二運モデル
- 大運・流年モデル
- 一般評価モデル
- 泰山流評価モデル

#### Data 層

- 設定永続化

- 「共通計算」「一般評価」「泰山流評価」「保留ルール」を分離管理すること

## 18. 受入基準

初版要件の受入基準は以下とする。

1. 入力情報から四柱・通変星・十二運・大運・流年が算出できること
2. 初心者向け表示と上級者向け表示の画面分離が可能であること
3. 一般四柱推命の共通計算ロジックと、泰山流評価ロジックがコード上分離されていること
4. SQLite による保存・読込ができること
5. PDF / テキスト / JSON の基本出力が可能であること
6. 要確認ルールを断定表示しないこと
7. 既知サンプルによる回帰テスト基盤を持つこと

## 20. まとめ

本プロジェクトの要点は、**命式計算の正確さ**と**評価ルールの分離設計**である。

MVP では、一般四柱推命として安定している共通計算基盤を 먼저完成し、その上に泰山流評価層を段階的に追加する。
これにより、不確実な流派固有ルールを推測で埋めずに、実装可能な共通基盤を先に確立できる。

以上を本プロジェクトの初版要件定義とする。
